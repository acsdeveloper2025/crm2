# CRM2 Audit — Independent Finding Verification Report

**Scope:** `/Users/mayurkulkarni/Downloads/crm2` only. **Date:** 2026-07-01. **Status:** Read-only re-verification — no code, config, or original audit report was modified.

## Methodology

Every finding from all 19 original per-domain audit reports (`docs/audit/01-authentication.md` through `docs/audit/19-mobile-api-compatibility.md`) was re-extracted fresh from the source `.md` files themselves — not trusted from `MASTER_REPORT.md`'s prior summary — yielding 63 findings. Each finding was then independently re-examined by re-reading the actual cited code/config/doc at the exact file:line in the live repository (not by re-reading the audit report's prose alone), tracing call chains where the claim depended on downstream behavior (e.g. whether a validation gap actually reaches an unguarded sink), and checking `docs/adr/`, `docs/COMPLIANCE_GAPS_REGISTRY.md`, `docs/DEFERRED_ITEMS.md`-equivalents, and inline code comments for evidence that a given risk was already knowingly accepted by the team rather than merely unaddressed. Each finding was classified as TRUE_POSITIVE, FALSE_POSITIVE, ACCEPTED_RISK, or INSUFFICIENT_EVIDENCE, with severity corrected where the original CVSS/impact calibration didn't match the actual verified blast radius and exploitability.

Duplicate detection was run as two independent clustering passes over all 63 findings, then reconciled by re-reading each proposed cluster's member findings side by side. Both passes converged on the identical 4 clusters (same canonical grouping, same member IDs), so the union and the intersection are the same set — no disagreement to arbitrate. No new findings were introduced anywhere in this process: this report only validates, corrects, and merges what the original 19 reports already contained.

## Duplicate Findings (Merged)

Both independent clustering passes converged on the identical 4 clusters (same canonical grouping, same member IDs, same rationale in substance) — no disagreement to arbitrate. The union of Pass 1 and Pass 2 is used as-is.

### MERGED: Missing HTTP security headers (CSP / X-Frame-Options / HSTS / Referrer-Policy / X-Content-Type-Options) in nginx and Express
- **Members:** XSS-01 (05-xss), API_SECURITY-01 (09-api-security), FRONTEND_SECURITY-02 (10-frontend-security), INFRASTRUCTURE-02 (13-infrastructure)
- **Reconciled disposition:** TRUE_POSITIVE (all 4 members agree)
- **Reconciled severity:** Medium (all 4 members independently landed on Medium/CVSS ~4.3–6.1; no conflict to resolve)
- **Merge rationale:** All four findings cite the exact same absence in the exact same two files — `infra/prod/nginx.conf` (zero `add_header` security directives in either `server{}` block; only `Content-Type`/`Cache-Control` at lines 30/65/116/121) and `apps/api/src/http/app.ts` (no `helmet` import, no manual security-header middleware). Four different audit lenses (XSS, API security, frontend security, infrastructure) independently discovered the identical single root cause. Treated as one finding for backlog purposes; the shared fix is one `helmet()` call plus a handful of nginx `add_header` lines.

### MERGED: Socket.IO CORS configured with `origin: true` + `credentials: true`
- **Members:** AUTHORIZATION-02 (02-authorization), CSRF-02 (06-csrf)
- **Reconciled disposition:** TRUE_POSITIVE (both members agree)
- **Reconciled severity:** Informational — the CSRF-02 audit's "Low" and AUTHORIZATION-02's "Informational" are reconciled in favor of the lower label because both members' own independent re-verification confirms the config is currently inert: the socket.io handshake requires a bearer JWT (not a cookie), so an attacker page reflected by the permissive CORS setting still cannot obtain a valid token to complete the handshake. No live exploit path exists today; this is latent config fragility, matching Informational.
- **Merge rationale:** Both findings cite the identical line, `apps/api/src/platform/realtime/index.ts:135-137` (`cors: { origin: true, credentials: true }`), and the identical mechanism. One is filed under Authorization, the other under CSRF — same code, same root cause, two audit angles.

### MERGED: Web access token stored in `localStorage`, readable by any XSS for its TTL (ADR-0076 SEC-10 accepted trade-off)
- **Members:** AUTHENTICATION-02 (01-authentication), FRONTEND_SECURITY-03 (10-frontend-security)
- **Reconciled disposition:** ACCEPTED_RISK (both members agree — both independently found the same explicit team acceptance: `docs/COMPLIANCE_GAPS_REGISTRY.md` SEC-10 and the `ADR-0076`-citing comment in `apps/api/src/http/refreshCookie.ts`)
- **Reconciled severity:** Low (both members agree)
- **Merge rationale:** Both cite the same file, `apps/web/src/lib/auth.ts` (lines 11-23 / 14-15, the same `access()` getter using `localStorage.getItem(ACCESS_KEY)`), describing the identical mechanism: ADR-0076 SEC-10 moved only the refresh token to an httpOnly cookie, deliberately leaving the short-lived access token in localStorage. Independently surfaced by the authentication audit and the frontend-security audit on the same code.

### MERGED: Unused root-level `vitest` / `@vitest/coverage-v8` devDependencies duplicating per-workspace declarations
- **Members:** DEPENDENCY_AUDIT-01 (16-dependency-audit), CODE_QUALITY-04 (17-code-quality)
- **Reconciled disposition:** TRUE_POSITIVE (both members agree)
- **Reconciled severity:** Informational — DEPENDENCY_AUDIT-01 was corrected from Low to Informational in this re-verification pass (see Severity Corrections above); CODE_QUALITY-04's original Low is superseded by that correction since both describe the identical root cause and should carry one consistent severity.
- **Merge rationale:** Both cite the identical file and lines (`package.json:32,43`), the identical root cause (root declares `vitest`+`@vitest/coverage-v8` devDependencies that duplicate what each workspace already declares, with no root script using them), both sourced from the same underlying `knip` finding. Pure dead-weight, zero functional/security impact.

## Net Effect on the Master Report

Of the master report's original 63 findings, **60 survive independent re-verification as real** (54 TRUE_POSITIVE + 6 ACCEPTED_RISK) and **3 are retracted** (SSRF-01, API_SECURITY-03, DEPENDENCY_AUDIT-03 — all FALSE_POSITIVE). **Zero findings remain open questions** (0 INSUFFICIENT_EVIDENCE) — every claim in all 19 reports was independently resolvable against the live repository without requiring live-system or production access.

After collapsing the 4 confirmed duplicate clusters (10 member findings → 4 canonical findings), the distinct actionable backlog is **54 findings** (down from the master report's nominal 63), none of which are Critical, one is High (LOGGING-03, the unbounded-Docker-log-growth / repeat-disk-exhaustion risk), and the remainder span Medium/Low/Informational.

Five severity corrections were made, all downward (Medium→Low ×3, Low→Informational, Medium→Low), because the original CVSS/impact calibration in those five cases did not match the actually-verified blast radius (pure documentation drift with no live exploit path, or a finding whose own text already conceded zero security exploitability). No severity correction moved a finding upward, and no new Critical or High findings were discovered. This shifts the master report's severity distribution slightly toward Low/Informational but does not add any new Critical/High risk.

**The original "READY FOR PRODUCTION" verdict still holds.** This independent re-verification found no Critical findings, exactly one High finding (LOGGING-03 — unbounded log growth, an operational-resilience gap with a known, already-once-materialized failure mode, not a security compromise or data-exposure vector), and confirmed that three findings the master report likely counted as open risk (SSRF-01, API_SECURITY-03, DEPENDENCY_AUDIT-03) are false positives that should be removed from the backlog entirely. The single High item (LOGGING-03) is the most actionable pre/post-launch item to prioritize — add Docker log-rotation (`max-size`/`max-file`) to `infra/prod/docker-compose.yml` — but its absence is an operational-hygiene gap with a known runbook-level mitigation (manual pruning, as already done once for the analogous image-disk-exhaustion incident), not a blocker to the READY FOR PRODUCTION conclusion.

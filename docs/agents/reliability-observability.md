# Ledger — Reliability & Observability Auditor (SRE)

Charter: AGENT_ORG.md row 9. Logging/metrics/traces · monitoring/alerting · idempotency · retries ·
graceful degradation · background-job reliability · rate-limiting · DR/backup/restore. SoT:
OBSERVABILITY_STANDARDS · MONITORING_STRATEGY · OPERATIONS_GUIDE + runbooks/ · DISASTER_RECOVERY ·
PERFORMANCE_STANDARDS (background jobs).

## Standing invariants
- **Graceful degradation:** a health/diagnostics path must NEVER 500 (precedent: `systemService.health()`
  must not call a throwing `loadEnv()` outside try). Degrade + report, don't crash.
- **Idempotency:** mobile writes carry `Idempotency-Key`; idempotent endpoints (start/complete/revoke/
  priority) return 409=success. Preserve this; never make a retried write double-apply.
- `@crm2/logger` is the only logger (6 levels, structured). No `console.*`.
- **>8s operations = background jobs** (exports, MIS, bulk assign, report gen, bank sync) with bell/toast/
  in-app completion; the user keeps working. Jobs must be retryable + observable.
- Backups ≠ recoverable: DR requires a tested restore (quarterly drill mandate in DISASTER_RECOVERY).

## Status — much infra PLANNED
Workers (`apps/worker`, `apps/report-worker`) are stubs; BullMQ/Valkey queues, monitoring/alerting, and
synthetics are not yet built. Until then: audit **readiness** (error-handling, idempotency, structured
logs, graceful-degrade in shipped code) + flag where reliability infra will be required.

This violates the standing "degrade, don't crash" lean for an operation the user explicitly asked for. Data-loss guard: the purge does NOT blindly destroy unsynced local work — PENDING attachments/form_submissions are marked `ABANDONED` (`:324`,`:335`), only SYNCED ones are deleted; the `tasks` row is hard-deleted. So a task reassigned away while the agent had un-uploaded field work loses that work — but that is the CORRECT semantic (the task is no longer theirs; a re-issue arrives as a fresh task with its own payout). At-most-once holds; no double-purge corruption.
  - **(3) ✅ Disjoint-set (cases vs revoked) holds — no upsert↔purge flip on the same task within a cycle.** The two server predicates are mutually exclusive on `ct.assigned_to`: `cases` = `ct.assigned_to = $1` (`repository.ts:85`); `revokedAssignmentIds` = `h.previous_assigned_to = $1 AND ct.assigned_to IS DISTINCT FROM $1` (`:118-120`). A row cannot satisfy `assigned_to = me` AND `assigned_to IS DISTINCT FROM me` in one snapshot → disjoint by construction. The reassigned-then-back case is correctly EXCLUDED from the delta (now `assigned_to = me` again) and flows in `cases` (verified vs the repository comment + writer). **TOCTOU note (benign):** `downloadForUser` and `revokedAssignmentIdsForUser` are two separate autocommit pool queries (no shared snapshot — grep confirms no `withTransaction` in sync). A task reassigned AWAY *between* the two page-0 queries could appear in BOTH (in `cases` from the earlier rows query, in the delta from the later one). The device processes upsert FIRST (`:84`) then purge (`:114`) → purge-wins → terminal local state = purged = the correct outcome. The reverse (reassigned TO me mid-request) cannot enter the delta (requires `assigned_to IS DISTINCT FROM me` at delta-time). So the race resolves to the right state regardless of ordering. No flip hazard.
  - **(4) ✅ Split-watermark cannot silently miss a purge — the strongest correctness point.** `cases` watermarks on `COALESCE(ct.updated_at, cs.updated_at) > cutoff`; the delta watermarks on `h.created_at > cutoff` — DIFFERENT columns. But both `assignTask` (REASSIGNED) and `unassignTask` (UNASSIGNED) are SINGLE transactions that `UPDATE case_tasks SET assigned_to=…, updated_at=now()` AND `INSERT task_assignment_history` in the same tx (`cases/repository.ts:607-646`, `:667-686`) → `ct.updated_at` and `h.created_at` share the transaction `now()`. The away-event therefore crosses BOTH watermark boundaries together; there is no skew window where the new assignee sees it in `cases` but the old assignee's delta misses it. Combined with watermark-only-advances-on-full-success (device `:166`), a persistently-failing cycle keeps the away-event `> cutoff` until a cycle lands. No lost purge from clock split or repeated failure.
  - **(5) ✅ Observability — delta-query failure IS surfaced, NOT silent.** `revokedAssignmentIdsForUser` is `await`ed inside `svc.download` (`service.ts:90`); a throw rejects → `controller.ts:16-18` `next(e)` → central handler logs `'unhandled error'` on the requestId child logger (`app.ts:122`) and the `finish` listener logs method/path/status/durationMs/userId (`app.ts:43-62`, `@crm2/logger`, no console.*). The device treats the failed page as `!response.success` → throws → cycle aborts WITHOUT advancing the watermark → retried next tick. Clean graceful-degrade: a delta failure fails the whole cycle (no partial apply), it does not half-purge. Query is bounded — migration 0057 adds `(previous_assigned_to, created_at)` matching the predicate exactly; with a live watermark the scan is bounded by sync cadence (30d floor only on first-ever sync, where there is nothing local to purge anyway).
  - **NOTE (non-gating) · delta is now a hard dependency of page-0 DATA delivery.** Because `revokedAssignmentIds` is computed in the SAME request that returns page-0 `cases`, a *persistently*-failing delta query (missing index in some env, pathological `task_assignment_history` growth) would 500 page 0 and thus block ALL down-sync for that device, not just purges — the agent stops receiving new tasks, not merely stops purging. Today this is acceptable (index shipped in 0057, query bounded, fail-closed is safer than serving cases while silently dropping the purge signal). CARRY for prod: if down-sync availability ever needs to survive a degraded delta, compute the delta in a try/catch that degrades to `revokedAssignmentIds:[]` + a `logger.warn` (purges defer one cycle) rather than failing the page — but ONLY if a missing purge-for-a-cycle is judged less harmful than a blocked sync, which for an assigned-away task (stale-but-harmless local row) it arguably is. Not gating; flag the trade for the prod hardening pass.
  - **Execution fields (`inProgressAt`/`completedAt`) — ✅ no reliability concern.** Emitted conditionally (omitted when null, `service.ts:36-37`, v1 wire-compat) and consumed through the device conflict resolver (`SyncDownloadService.ts:512-526` → `mergedState.inProgressAt`/`.completedAt`), which preserves local pending state via `local_updated_at`/`sync_status` (the D1 fix) so a server re-emit after a local wipe re-hydrates timestamps without clobbering fresher local work. No double-apply, no overwrite of in-flight local edits.
  - **Bottom line:** PASS. offset-0-only is safe (fixed-watermark multi-page loop + full-cycle-only watermark persist + idempotent purge ⇒ no lost purge across crashes/pages); the purge is at-most-once by row-absence and abandons-not-silently-destroys unsynced work (by design); the cases/revoked sets are disjoint by construction and the only TOCTOU resolves purge-wins = correct; the split-watermark hazard is closed because the reassign writer stamps `ct.updated_at` and `h.created_at` in one tx; delta failures fail-closed and are structured-logged. **No FLAG/BLOCK.** One NOTE (delta is now a hard dependency of page-0 data — consider degrade-to-`[]` for prod sync availability) + the minor doc imprecision that purge idempotency is row-absence, not `recentlyCleaned` (which only guards re-hydration).

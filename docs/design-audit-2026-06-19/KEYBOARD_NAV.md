# D15 — Keyboard Navigation

**Totals (deduped):** 13 P1 · 38 P2 · 54 P3. Skeptic re-verified 18 verdicts; **5 distinct P1s confirmed at source**; 2 findings refined down (see below).

## ⚠️ Skeptic refinement (don't over-fix)

- **Admin CRUD cluster — D15 P2 "Focus indicator (global)" framing that .btn/.btn-ghost have N** — Imprecise (severity-affecting for the Operations P1). There IS a global app-defined focus ring: packages/ui-theme/src/tokens.css:238-240 declares `:focus-visible { outline: 2px solid hsl(var(--ring)); outline-offset: 1px }` in @layer base. .btn/.btn-ghost (index.css:14-19) do NOT set outline-none, so they INHERIT this 2px brand ring — they are not on the bare UA default and are clearly focus-visible. Only `.input` (index.css:12) sets `outline-none focus:border-ring`, which (being in @layer components, ranked above base) suppresses the ring for TEXT INPUTS only, leaving a 1px border-color shift. So the real defect is input-only, not "every .btn/.btn-ghost control." The Settings cluster got this exactly right ("Buttons (.btn/.btn-ghost) do NOT override outline, so they keep the proper focus ring"); the Admin CRUD and Operations clusters overstated it.
- **Operations cluster — D15 P1 "Focus visibility (global, affects every control on all 5 page** — Overturned as P1, and the "no global :focus-visible rule" sub-claim is FALSE. A global :focus-visible 2px ring exists in tokens.css:238-240 (imported by index.css:1) and applies to all buttons/links/bucket buttons — they ARE keyboard-focus-visible. The genuine residual defect is the input-only suppression (.input outline-none), which is at most P2/P3 (low-contrast 1px border shift on text inputs), not a P1 "no visible focus ring" on every control. No keyboard action is unreachable due to focus visibility, so it does not meet the P1 bar (action unreachable / keyboard trap).

**Net:** a global `:focus-visible` 2px ring DOES exist (`packages/ui-theme/src/tokens.css:238-240`) and applies to buttons/links — so the real focus-visibility defect is **text-`.input` only** (it sets `outline-none`), a P2/P3, not an app-wide P1.

## The 5 confirmed P1 keyboard blockers

1. **DataGrid sortable headers are keyboard-unreachable** — `<th onClick>` with no `tabIndex`/`role`/`onKeyDown` (`DataGrid.tsx:681-695`). No grid in the app can be sorted by keyboard. → render the header label as a `<button>` (or `role=button tabIndex=0 onKeyDown`).
2. **DataGrid row-click navigation is keyboard-unreachable** — `<tr onClick>` with no keyboard equivalent (`DataGrid.tsx:773-784`). Can't open a case/detail from any list via keyboard. → make the primary cell a real link, or `tabIndex=0 role=button onKeyDown` on the row.
3. **Commission Rates dialog has no focus trap** (`CommissionRatesPage.tsx:119-125`) — Tab leaks behind, Escape inert. (Also a known a11y P1; moot once it goes inline per D14.)
4. **RBAC "Cannot deactivate" overlay** is a bare `<div>` with no role/aria-modal/focus-trap (`features/access/RolesPage.tsx:224-235`) — focus stranded, Escape doesn't close.
5. **RateManagement `SearchableSelect` commits via `onMouseDown` only** (`RateManagementPage.tsx:78-115`) — no key handler/combobox roles → keyboard users can filter but cannot select, blocking the inline Add-Rate flow + toolbar filters.

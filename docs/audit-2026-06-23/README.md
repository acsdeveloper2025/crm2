# Mobile Verification Round-Trip Audit — 2026-06-23

**Scope:** end-to-end trace of every CRM2 verification type through the 6-hop round-trip (Create → Down-Sync → Form → Up-Sync → Report → web render), adversarially verified against the live `crm2_dev` DB, the crm2 API/SDK/web, and the `crm-mobile-native` v2-native contract (ADR-0054). Read alongside the machine-readable [`findings.json`](./findings.json) (the authoritative record — all 153 confirmed findings verbatim) and the draft [`proposed-registry-section.md`](./proposed-registry-section.md).

> **Authoritative artifact:** `findings.json` (153 raw confirmed findings across 12 tracks). This README is the human rollup + a deduped 80-row findings table. The orchestrator merges the registry section after review; this audit changed **no source code**.

---

## 1 — Executive summary

A 4-lens (CEO / CTO / Designer / Security) adversarial trace of all 12 tracks surfaced **153 confirmed findings** (raw), deduping to **80 distinct findings** once true cross-lens / cross-track duplicates are merged. **No CRITICAL** issues; **11 HIGH** (deduped). The dominant themes:

1. **Report split-tenure / key-drift (the single biggest cluster).** The field-report **raw "sections" view is built on un-canonicalized `form_data`** (`fieldReports/service.ts:28` `buildSections(ctx.formData)`), while the period-recombine + outcome-label logic lives only in the **narrative** path (`canonicalize.ts`, wired at `service.ts:55`). Because the v2-native device emits every tenure **split** as `<base>Value`+`<base>Unit` and the sectionMap references the **combined** key, every named "…Period" row renders empty and the split halves spill into a generic "Additional Details" catch-all — on **every field type**. Compounded by per-type ref drifts: RESIDENCE/RCO read `applicantStayingFloor` but the device emits `addressFloor`; BUSINESS reads `approxArea` but the device emits `officeApproxArea`. Only **RESIDENCE** has an active FIELD_REPORT layout, so for the other 8 field types the raw sections view **is** the entire report — the drift is 100% user-visible there.

2. **Server-driven outcome catalog is dead on mobile (all 9 field types).** `setOutcomesFromSync` keys on **short** codes (`RV/OV/PAV…`) while the reference endpoint + DB emit **long** codes (`RESIDENCE/OFFICE/PROPERTY_APF…`), so every synced outcome row is dropped and the device permanently falls back to compiled-in defaults. Admin re-label/re-order/add/remove of outcomes never reaches devices. This is also the **enabler of the PROPERTY_APF `NEGATIVE` bug**: NEGATIVE is offered as a top-level outcome with no APF form and no narrative branch, yielding an empty APF narrative.

3. **Attachment integrity is captured but never surfaced, and one IDOR.** `hash_verified`/`client_sha256` transit-tamper signal is computed, stored, logged-on-mismatch — and **never read** by any web/report/sync path, so a tampered photo is indistinguishable from a verified one. Separately, the **field-photo upload idempotency-replay** lookup is **unscoped** (global `operation_id`), leaking another agent's GPS + storage-key/filename (IDOR, mitigated only by client UUID randomness). The **FIELD_REPORT route is case-grain not task-grain** (a field agent can read a sibling task's full PII), **FIELD_AGENT PINCODE/AREA EXPAND** broadens `case.view` to other agents' cases, and **case-level KYC documents (PAN/Aadhaar) are servable/deletable cross-scope** because the attachment route skips the `caseVisible()` precondition.

4. **KYC desk-path gaps.** The uploaded KYC document (the actual evidence) **never appears in any client report**; **`required_attachments` (≥1 DOCUMENT) is not enforced at completion** (a KYC verdict can be finalized with zero documents); and **no guard binds unit `kind` to the operator-chosen `visitType`**, so a KYC unit can be mis-routed FIELD and bricked on a device with no KYC form.

5. **Revoke integrity.** `bulkAssign` **re-points a live ASSIGNED task in place** with no revoke + no mandatory reason (ADR-0055 bypass; this is the known registry SHIP-1, built-not-pushed); down-sync **masks a server REVOKED status** when local has unsynced edits, skipping the PII wipe and leaving the revoked form editable; and the revoke reason is **free-text end-to-end**, never canonicalized to the 9 `revoke_reasons` (v1's structured `task_revocations` record was lost).

**Live-data caveat:** 8 of 9 field types have **no active FIELD_REPORT layout** (only `RESIDENCE` + the `CASE_REPORT`), so most narrative-path findings are **latent until a layout is activated** — but the raw-sections findings are live on every submission, and RESIDENCE narrative findings are live today.

## 2 — Canonical matrix (live `crm2_dev` DB, 2026-06-23)

- **9 FIELD_VISIT types** (`kind=FIELD_VISIT`, `is_system=true`, `required_photos=5`, `required_gps=true`, `report_template_type=FIELD_NARRATIVE`, each has a `required_form_code`). Outcomes (`verification_unit_outcomes`):
  - `RESIDENCE`, `OFFICE`, `RESIDENCE_CUM_OFFICE`, `BUSINESS`, `BUILDER`, `NOC`, `DSA_CONNECTOR` → **5 each**: `POSITIVE, SHIFTED, NSP, ENTRY_RESTRICTED, UNTRACEABLE`
  - `PROPERTY_APF` → **4**: `POSITIVE, NEGATIVE, ENTRY_RESTRICTED, UNTRACEABLE`
  - `PROPERTY_INDIVIDUAL` → **4**: `POSITIVE, NSP, ENTRY_RESTRICTED, UNTRACEABLE`
  - = **43 type × outcome pairs**
- **59 KYC_DOCUMENT types** (ids 10–68, `is_system=false`, photos=0, no GPS, no form code, `report_template_type=KYC_DOCUMENT`), `result_set={Positive,Negative,Refer,Fraud}`. Desk/relay path (KYC_VERIFIER two-actor); document upload; **no mobile field form**.
- **9 `revoke_reasons`**: `NOT_MY_AREA, WRONG_PINCODE, ADDRESS_NOT_WORKING, NOT_WORKING, CUSTOMER_LEFT_AREA, LEFT_AREA, WRONG_ADDRESS, WRONG_INCOMPLETE_ADDRESS, OTHER`.
- `report_templates` table is **EMPTY**; narrative comes from `packages/sdk/src/fieldReportDefaults.ts` + `apps/api/.../fieldReports/sectionMap.ts` + `canonicalize.ts`. Only active `report_layouts`: **id1 FIELD_REPORT/RESIDENCE**, **id2 CASE_REPORT**. So **8 of 9 FIELD types have no active report layout** (narrative latent until activated).

**The 6 hops traced** (any field renamed / reshaped / dropped / mislabeled / unit-changed / mis-rendered flagged):
1. **CREATE** (web→API→DB) · 2. **DOWN-SYNC** (DB→mobile, bare v2 envelope, ADR-0054) · 3. **FORM** (mobile builders + outcome-conditional fields, ≥5 photos+GPS+selfie) · 4. **UP-SYNC** (mobile→API→DB, camelCase `form_data` + `client_sha256`) · 5. **REPORT** (DB→canonicalize→sectionMap→render) · (web render of Field Report + Field Photos GPS overlay).

## 3 — Methodology

For each of the **12 tracks** (RESIDENCE, OFFICE, RESIDENCE_CUM_OFFICE, BUSINESS, BUILDER, NOC, DSA_CONNECTOR, PROPERTY_APF, PROPERTY_INDIVIDUAL, KYC, REVOKE, CROSSCUT): **trace** the fields across all 6 hops → apply **4 lenses** (CEO product-impact, CTO correctness/contract, Designer UX/labeling/grouping, Security IDOR/PII/integrity) → **adversarially verify** each candidate against source + live DB (refuted findings already removed) → **synthesize**. Every finding carries `evidence` (file:line + DB proof) and a `verifyConfidence`. This synthesizer step **deduped true duplicates** (same defect at the same file:line raised by multiple lenses/tracks) into 80 rows while preserving every track and listing all source ids + lenses + affected tracks per merged row.

# ADR-0031 — OpenAPI surface + SDK-drift gate (validate, don't replace)

- **Status:** Accepted
- **Date:** 2026-06-15
- **Deciders:** CTO (owner-delegated)
- **Supersedes / amends:** none (additive — fulfils B-11/B-12; activates Part 21 of `docs/CI_CD_STANDARDS.md` and the OpenAPI/`DOCUMENTATION_AS_CODE.md` intent under ADR-0011).

## Context

`DOCUMENTATION_AS_CODE.md` + ADR-0011 declare OpenAPI the documentation source of truth and call for
a contract-drift gate (Part 21). Until now there was no spec and the CI step was a stub. Meanwhile we
have **one hand-written `@crm2/sdk`** consumed by web AND the mobile app (~133 typed methods), with no
automated check that it still matches the server.

Two ways to build the gate, and they differ enormously in blast radius and quality:

1. **Generate the SDK from OpenAPI** (replace the hand-written client). Rejected. A generated HTTP
   client mainly pays off for many external consumers in many languages; we have **one internal TS
   SDK**. Generation would mangle the deliberately-shaped transport ergonomics (`reqBlob`/`apiExport`,
   the array-vs-`Paginated<T>` envelope discipline, mobile byte-compat per ADR-0012), worsen DX and
   tree-shaking, force spec-first YAML authoring against a code-first zod reality, and reopen a frozen
   decision — a **downgrade** that buys no drift safety the validate gate doesn't already give.

2. **Validate, don't replace.** Keep the hand-written SDK authoritative; emit OpenAPI *from the code*
   and gate on drift. Chosen.

The structural insight: drift-freedom comes from a **single source of truth that everything derives
from** — not from code-generating a client. The genuinely-best end state is **zod as the contract
source** (request *and* response schemas in zod → server validates against them → OpenAPI generated
from them → SDK types `z.infer`'d from them → drift is impossible, not merely caught). Today request
schemas are **already shared zod** (the server imports the same `Create*Schema` the SDK's input types
are inferred from), so request drift is already structurally prevented; responses are still TS
interfaces. Moving responses into zod across all modules is a migration, not a cluster-tail item.

## Decision

Ship **phase 1 = the API SURFACE contract + drift gate**, explicitly as step 1 of the zod-single-source
trajectory (not a detour):

1. **Generator** (`platform/openapi/document.ts`) — `buildOpenApiDocument(app)` derives an OpenAPI
   3.1 document from the **live Express app** (the single source of truth for what endpoints exist):
   paths · methods · tags (first path segment) · security (Bearer-JWT by default, cleared for the four
   public endpoints `/health`, `/time`, `/auth/login`, `/auth/refresh`). Pure `app → doc`, DB-free.
   Route enumeration walks the router tree with **zero new dependencies** (Express-4 string mounts
   compile to a deterministic regexp we parse for the prefix). Request/response **schemas are phase 2**
   (omitted, not forgotten — the `info.description` and operation `responses` say so).

2. **Committed artifact** — `apps/api/openapi.json`, regenerated by **`pnpm openapi`**
   (`platform/openapi/cli.ts`), deterministic `JSON.stringify` (prettier-ignored). It is the
   human/external/future SoT (B-11).

3. **Drift gate (B-11/B-12), two complementary mechanisms — no rotting allowlist:**
   - **Surface drift (CI Part 21 + the contract test):** re-emit and assert the committed
     `openapi.json` equals the freshly-built doc. Any route added/removed/renamed that wasn't
     regenerated fails — covering **every** route (incl. import/bulk/export/dashboard). The contract
     test enforces it in the normal `pnpm test` suite too, not only in CI.
   - **SDK→route parity (contract test):** every path the hand-written SDK references must resolve to a
     real mounted route — the SDK can never point at a dead/renamed endpoint. This is the high-signal,
     zero-false-positive direction. The **reverse is intentionally NOT asserted**: the web app calls
     many endpoints (import/bulk/export/dashboard/photo) via the `api()` helper rather than the typed
     client, so those routes legitimately have no SDK method; asserting it would mean a ~40-entry
     allowlist that rots. Surface coverage of those routes is fully handled by the OpenAPI git-diff.

## Consequences

- **Positive:** drift is caught mechanically (the class behind the array→envelope trap and the stale
  `notifications.test.ts`); the SDK stays hand-written, authoritative, and ergonomic; no frozen-decision
  reopen, no new dependency, no big-bang rewrite; an OpenAPI artifact now exists for humans/tooling.
- **Negative / accepted:** phase-1 operations carry no request/response schema bodies yet (the spec
  describes *which* endpoints exist + auth, not the payload shapes). Response-shape drift is **not**
  caught until phase 2; request-shape drift is already prevented by shared zod.
- **Phase 2 (future, the stated direction):** move responses into zod schemas (in `@crm2/sdk`), validate
  them server-side, emit them into the spec via `@asteasolutions/zod-to-openapi` (a dependency added
  *then*, when it earns its place), and `z.infer` the SDK response types — at which point the spec is
  100% generated and drift becomes structurally impossible. Each module can adopt incrementally.

## Alternatives considered

- **Generate/replace the SDK** — rejected (above): downgrade for a single-consumer internal TS SDK.
- **Hand-write the OpenAPI YAML** — rejected: a second hand-maintained artifact = a new drift source.
- **Assert full route↔SDK bijection now** — rejected: a large, rotting exemption list for the
  deliberately-`api()`-called endpoints; low signal. The git-diff covers those routes instead.
